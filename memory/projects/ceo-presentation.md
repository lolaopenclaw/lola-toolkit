- **Contenido:** Transición PO → AIPM, 8 principios, toolkit, 4 casos de estudio reales

### 2. Karpathy — "La app eres tú"
- **Fuente:** https://youtu.be/kwSVtQ7dziU
- **Ideas clave:**
  - El software está desapareciendo como producto → la IA ES el producto
  - Los agentes no son herramientas, son una comunidad de investigación
  - El PO pasa de gestionar backlog a orquestar agentes
- **Útil para:** Visión de futuro, justificar el cambio de paradigma

### 3. Javier Garzás — 7 Pecados del Vibe Coding
- **Fuente:** https://youtu.be/Yn59vQQAJpI
- **Autor:** Doctor en Ing. Informática, pionero agilidad España-LATAM, 20+ años, +600 empresas
- **Los 7 pecados:**
  1. No marcarle límites a la IA (guardarraíles)
  2. Equipo vulnerable (no saben mantener lo que crean)
  3. No tener método de trabajo
  4. No usar specs
  5. Specs gigantes (volver al waterfall)
  6. No hacer testing específico para vibe coding (LLM como juez, TDD adaptado)
  7. **No usar dos carriles** (vibe coding lane + engineering lane en paralelo)
- **Validación:** 6/7 respaldados por industria global. El "dos carriles" es síntesis original de Garzás combinando Dual-Track Agile + consenso emergente
- **Útil para:** Framework de errores a evitar, argumentar que se necesita gestión profesional (= PO evolucionado)

### 4. Tendencias de la industria
- **SDD (Spec-Driven Development):** AWS lo lanzó como producto (Kiro), GitHub sacó Spec Kit (sept 2025), InfoQ, DevLand 2026. Es EL framework emergente.
- **IBM:** "Objective-Validation Protocol" — el software pasa de vibe coding a validación estructurada
- **Forrester (Lo Giudice):** "Vibe coding → Vibe Engineering antes de fin de 2026"
- **Forbes:** Empresas que usan vibe coding mueven concepto → prototipo más rápido, confirman dirección antes de ingeniería seria
- **Andrew Ng:** Critica el término "vibe coding" porque engaña — los profesionales no van "con las vibes"
- **The New Stack:** "From vibes to engineering: How AI agents outgrew their own terminology"

### 5. Experiencia propia (Manu + Lola)
- 11 subagentes en paralelo en una tarde
- Multi-agent architecture con templates especializados
- Guardarraíles implementados (SOUL.md, AGENTS.md, verification protocol)
- Specs como templates de agente
- Testing integrado (autoimprove eval, verification-before-completion)
- **Útil para:** Demostración práctica de que un PO ya puede orquestar agentes productivos
